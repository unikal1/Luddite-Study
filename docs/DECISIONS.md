# 결정 기록

## 0001. Vite + React + TypeScript 사용

상태: 확정

대시보드, 탐색기, 작성 미리보기, 운영 도구처럼 상호작용이 많은 정적 앱이므로 Vite + React + TypeScript를 사용한다.

## 0002. GitHub Pages 정적 배포

상태: 확정

별도 백엔드 없이 GitHub Pages로 배포한다. 저장소명 기준 경로는 `/Luddite-Study/`이며 Vite `base`에 반영한다.

## 0003. Markdown + YAML frontmatter와 JSON을 정본 데이터로 사용

상태: 확정

본문 중심 자료는 Markdown으로 관리하고, 사용자/회차/진도/벌칙 같은 운영 메타데이터는 JSON으로 관리한다.

## 0004. 브라우저에서 GitHub write API와 OAuth를 사용하지 않음

상태: 확정

GitHub Pages에서 토큰을 안전하게 숨길 수 없으므로 브라우저 직접 저장 기능은 제외한다. 작성 화면은 초안, 미리보기, 추천 경로, 복사, GitHub 링크만 제공한다.

## 0005. Issues/Discussions는 핵심 저장소로 쓰지 않음

상태: 확정

Issues/Discussions는 토론과 운영 기록 링크로만 사용한다. 화면에 표시되는 정본 데이터는 저장소 파일이다.

## 0006. 초기 샘플 데이터

상태: 확정

초기 사용자는 `alice`, `bob`, `chris`이고 회차는 1-3회차로 둔다. 샘플 자료, 발표, 일정, 목표, 진도, 벌칙은 실제 운영 데이터와 같은 구조를 사용한다.

## 0007. 수정/삭제도 정적 UX로 처리

상태: 확정

브라우저가 저장소에 직접 쓰지 않는 원칙을 유지하면서 기존 문서 수정과 삭제 준비 플로우를 제공한다.

결과:

- 수정은 기존 Markdown을 선택해 폼에 불러오고, 수정된 Markdown을 복사한 뒤 GitHub edit 화면에 붙여 넣는다.
- 삭제는 대상 문서를 선택해 삭제 체크리스트를 만들고, GitHub에서 파일 삭제와 참조 정리를 수행한다.
- 삭제 시 `data/sessions.json`의 `resources` 같은 참조를 함께 확인하도록 안내한다.

## 0008. 데이터와 콘텐츠 참조를 빌드 시 검증

상태: 확정

정적 사이트는 배포 후 서버 검증이 없으므로 빌드 시점에 JSON과 Markdown 참조 오류를 실패시킨다.

결과:

- `src/data/validate.ts`가 JSON 필드, 상태값, 중복 ID, 사용자/회차 참조를 검증한다.
- `src/contentValidation.ts`가 Markdown owner, 발표 session, 회차 resource 참조를 검증한다.
- 검증 실패는 unit test와 production build에서 드러난다.

## 0009. 운영 도구 초안은 대상 파일 링크와 함께 제공

상태: 확정

운영자가 JSON 조각이나 회의록 Markdown을 복사한 뒤 어느 파일에 반영해야 하는지 바로 알 수 있어야 한다.

결과:

- 다음 일정, 진도, 벌칙 초안은 각각 `data/sessions.json`, `data/progress.json`, `data/penalties.json` 대상 경로를 표시한다.
- 각 초안은 대상 파일의 GitHub edit 링크를 제공한다.
- 회의록 초안은 추천 Markdown 경로와 GitHub 새 파일 생성 링크를 제공한다.

## 0010. 브라우저 검증은 주요 플로우와 접근성을 함께 본다

상태: 확정

정적 사이트는 배포 후 런타임 문제가 발견되기 쉬우므로 Playwright로 실제 화면을 열고 axe 접근성 스모크 테스트를 같이 실행한다.

결과:

- 데스크톱과 모바일 viewport에서 대시보드, 자료 탐색, 발표 탐색, 작성, 운영 도구를 검증한다.
- dashboard와 composer에는 자동 접근성 검사를 실행한다.
- GitHub Actions 배포 workflow도 동일한 E2E 검증을 통과해야 한다.

## 0011. 수정 모드는 기존 경로를 보존

상태: 확정

기존 Markdown을 수정할 때 문서 종류, 사용자, 회차, 파일명을 화면에서 바꾸면 실제 파일 경로와 frontmatter가 어긋나 배포 검증이 실패할 수 있다.

결과:

- 수정/삭제 모드는 기존 문서 경로를 기준으로 동작한다.
- 수정 모드에서는 문서 종류, 사용자, 회차, 폴더, 파일명 입력을 잠근다.
- 파일 이동은 GitHub에서 경로와 frontmatter를 함께 바꾸는 별도 작업으로 처리한다.

## 0012. Markdown 이미지는 public/assets 절대 경로만 허용

상태: 확정

Markdown 파일은 raw 문자열로 번들링되므로 문서 옆 상대 이미지 파일은 Vite 산출물에 자동 포함되지 않는다.

결과:

- 이미지 파일은 `public/assets/`에 둔다.
- Markdown에서는 `/assets/...` 경로로 참조한다.
- 상대 이미지 경로는 `src/contentValidation.ts`에서 빌드 실패로 처리한다.

## 0013. 운영 초안은 안전한 값으로 보정

상태: 확정

운영 화면에서 복사한 JSON 조각은 그대로 `data/*.json`에 들어갈 수 있어야 하므로 음수 진도, 음수 벌칙 금액, 빈 일정/목표 같은 값을 안전한 기본값으로 보정한다.

결과:

- 진도는 `0-target` 범위로 보정한다.
- 벌칙 금액은 `0-1,000,000` 범위로 보정한다.
- 빈 날짜, 시간, 목표, 사유는 검증 가능한 기본값으로 보정하고 화면에 안내한다.

## 0014. 기본 화면은 핵심만 보이고 세부는 접는다

상태: 확정

운영 도구는 기능이 많지만 한 화면에 모든 데이터와 JSON 원문을 보여주면 진행자가 즉시 판단하기 어렵다.

결과:

- 대시보드는 현재 회차, 오늘 볼 것, 운영 상태, 최근 변경 자료만 기본 노출한다.
- 운영 도구는 데이터 상태와 발표자 뽑기를 먼저 보여주고, 변경 초안은 `일정`, `진도`, `벌칙`, `회의록` 중 하나만 보여준다.
- JSON/Markdown 원문은 `초안 보기`를 열 때만 노출한다.
- 작성 화면의 폴더, 파일명, 태그는 `세부 설정` 아래로 접는다.
- 삭제 준비 모드는 대상 선택과 체크리스트만 보여준다.
